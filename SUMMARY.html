The Flaky Network — Architecture Summary

This document describes the sender/receiver implementation submitted for the assignment: a real-time media transport that carries one 160-byte frame per 20ms across a lossy, jittery, reordering, duplicating UDP link, with playout on a steady 20ms clock at the far end.

1. Problem framing

A lost packet in a real-time stream has exactly two families of remedy: a reactive one (detect the loss, ask for a resend) or a proactive one (send enough extra information ahead of time that the loss doesn't need to be reported at all). Reactive recovery costs at least two network transits — the request out, the resend back — which at a 20ms frame cadence on a jittery link routinely exceeds the deadline before the resend even arrives. Proactive recovery costs bandwidth instead of time, which is exactly what the 2.0x overhead budget is for. This implementation is entirely proactive: no retransmission, no NACKs, no use of the optional feedback path.

2. End-to-end architecture

harness source --47010--> SENDER --47001--> relay --47002--> RECEIVER --47020--> harness player

The sender speaks the fixed harness format on its input leg (47010) and a custom wire format on its output leg (47001, through the relay to the receiver's 47002). The receiver re-emits the fixed harness format on its output leg (47020). The feedback path (47003/47004) is unused.

4. Sender design

  • Single-threaded blocking read loop on the harness input socket (47010) — the harness itself paces frame delivery to exactly one every 20ms, so the sender never needs its own clock.
  • Keeps only the single most recent frame's payload in memory as the "previous frame" candidate for piggybacking — no larger window is needed since the receiver only ever needs to recover the immediately preceding frame from redundancy.
  • Piggybacks the previous frame on every outgoing packet except when seq % 16 == 0. This keeps the average packet size at roughly 1.97x the raw 160-byte frame — under the 2.0x budget with margin — while still covering the overwhelming majority of frames with a spare copy.

5. Receiver design

  • Jitter buffer: a fixed-size ring buffer of 512 slots, indexed by seq % 512, each slot storing the payload and the seq that last wrote it (so stale data from a wrapped-around slot is never mistaken for current data). At 20ms/frame this gives roughly 10 seconds of reorder/jitter headroom, comfortably larger than any playout delay this assignment would reasonably use.
  • Redundancy recovery: when a packet arrives carrying a redundant copy of frame (seq−1), the receiver fills that slot from the redundant copy only if it isn't already filled — so a legitimately arrived primary copy is never overwritten by a stale redundant one, and a duplicate delivery of either copy is a no-op.
  • Duplicate handling: because slots are keyed and checked by seq before being written, packets the relay delivers twice (or redundant copies that arrive after the primary already did) are simply ignored — no separate dedup table needed.
  • Playout clock: a single loop computes each frame's deadline as T0 + delay_ms + i*20ms and uses select() with a computed timeout to sleep until that deadline while still being able to absorb any packets that arrive in the meantime. At the deadline, whatever is in the buffer for that frame index is emitted to the harness player on 47020 in the harness's required format; if nothing is there, the frame is skipped — a miss, but the receiver does not stall or block waiting for it.

6. Why this hits a low delay

Because loss recovery here costs no extra time — the redundant copy travels alongside the very next frame, one network transit after the original — the jitter buffer only has to absorb ordinary delay variance (jitter and reordering), not the round-trip time of a retransmission request. That is what allows delay_ms to be tuned much lower than a NACK-based design could achieve.

7. Known limitation

A frame is unrecoverable only when both its primary packet and the one packet carrying its sole redundant copy are lost — i.e., two consecutive losses in the same burst. Single, isolated losses (the common case on a jittery-but-not-bursty link) are fully recovered at no added latency. This is the direct tradeoff of a depth-1 redundancy scheme chosen to fit the 2.0x bandwidth ceiling; a deeper redundancy window (piggybacking N previous frames instead of 1) would tolerate longer bursts at proportionally higher bandwidth cost.

8. Buffer ownership and byte order (systems-level notes)

  • All multi-byte header fields are sent in network byte order (htonl/ntohl) to match the harness's own framing convention.
  • Each frame's payload is copied into fixed-size stack/static buffers at the point of receipt; no pointers into socket receive buffers are retained across iterations, so there is no aliasing between one packet's data and the next.
  • The receiver's playout loop never blocks indefinitely: select() is always given a bounded timeout computed from the next deadline, so a stalled network cannot prevent the process from continuing to hit its 20ms playout cadence.
